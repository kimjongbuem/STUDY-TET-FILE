정규화 -> insert, delete, update, anormory  한개의 row에는 중복되는게 없어야한다. 중복이 있다면 유지보수 문제, 해당 속성의 추가/삭제문제 등..
그렇기때문에 더작은 스키마로 나누는 정규화 과정이 필요하다. 분해를 잘 못하면 정보의손실(손실 분해 <-> 비손실분해)
정규화에서 필요한 조건은 다음과같다.
1. cell Atomic 2. 해당 속성이 내부적으로 구조를 갖지 않는 형태, 분리가능한 형태 (Composition, multivalue) 허용 xxx

정규화의 단계는
1NF -> 2NF -> 3NF -> BCNF -> 4NF -> 5NF 정도가 있다 대충 정규화가 괜찮은 디비라면 4,5NF 형은 잘 안다룸.

1NF ) Composition, multivalue를 제거하자. 
함수종속(legal relation)을 이용한 분해 
SUPER KEY 가 모든것을 결정 -> 아토믹. 
PRIMARY KEY(A) -> (B,C) -> (D) // A가 B,C 결정 , B,C가 D결정..
EX) 사실의 모습 ID-> NAME, dept_name .. dept_name->building 
또한 함수종속성은 원자성이 원칙이기에 . 같은 속성값이 있다면 원자성이없으나(not hold) 역으로 반대로 가서 다른 속성값이라면 아토믹(hold)
=> 중복된 투플의 결과를 나타낼수없다 EX ) NAME 이 같은 투플의 경우....

2NF 기본키중에 특정 컬럼에만 종속된 컬럼(부분적 종속)이 없어야 한다

key a , b, c가 r(a~e)결정시 a,b,c-> r  ok!!! 이 테이블은 후보키얌!
결정자 id, 결정자 reference -> BCNF 
추론과 결과 , 인의적인 상호종속성 파악과 유추 -> F+ closure 폐포 ==> 주어진 집합 f로부터 추론될 수 있는 모든 함수 종속의 집합을 의미함. 
자명(trivial) : 일반적으로 a -> b , b<= a 


3NF. 슈퍼키가 아니여도 상관없음.
이행적 함수 종속이 없는 것 입니다. 풀어서 말하자면, 기본키 이외의 다른 컬럼이 그외 다른 컬럼을 결정할 수 없는 것입니다. 
릴레이션 스키마 R과 관련된 함수 종속 집합 F에서 A <= R B<= R일 때 A->B  형태를 갖는 F+의 모든 함수 종속들이 다음 들 중 적어도 하나를 보존한다면 3차 정규형이다. 
A->B 는 명백한 함수 종속이다, A는 R의 슈퍼키이다., B-A에 속한 각각의 속성 A는 R의 후보키를 갖고 있다.

BCNF 보이스 코드 정규형(Boyce codd Normal Form) BCNF란 3차정규형을 만족하면서 모든 결정자가 후보키 집합에 속한 정규형
BCNF에서는 A->B의 형태로 존재함 단 A는 무조건 슈퍼키.
